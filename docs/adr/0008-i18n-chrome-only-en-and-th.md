# i18n is chrome-only across EN and TH

We're adding Thai alongside English so members of a Thai-speaking trip group can use the app in their native language. The translation is deliberately limited to UI chrome — nav, buttons, form labels, empty states, page titles. The domain glossary defined in `CONTEXT.md` (Pot, Slip, Receipt, Contribution, Fronted Expense, Reimbursed, Fair Share, Net Position, Member, Admin) stays English in both locales. Those terms have explicit "_Avoid_" alternates in `CONTEXT.md` precisely because they are load-bearing terms of art for the group; translating them would either invent Thai equivalents the team doesn't actually use ("กระปุก" for Pot?) or pick loanwords that drift from the canonical English ("สลิป" for Slip is fine, but "Net Position" has no clean Thai equivalent). Doubling the glossary now without a product driver multiplies the design surface for no win.

Three further deliberate v1 narrowings, each pragmatic rather than principled: **(a)** form validation errors and server-action error strings (`'Required'`, `'Invalid amount'`, `'Only admins can log pot expenses'`, etc.) stay English — most are admin/edge-case messages, and translating them requires either a Zod errorMap refactor or threading `getTranslations()` through every server action; **(b)** currency and date formatting are untouched — `formatBaht` keeps its `en-US` thousand separators and `฿` prefix; dates keep `toLocaleDateString()` which picks the browser locale, not the app locale (so a Thai-browser user in EN-app mode sees Thai-formatted dates — a small inconsistency we accept); **(c)** Supabase auth emails stay English because they're sent from outside the app and aren't aware of the locale cookie.

Mechanism: `next-intl` with **cookie-only routing** (no `app/[locale]/` refactor, no URL prefix). The default for users with no cookie is **English**. Preference is stored only in the `NEXT_LOCALE` cookie — no `users.locale` column — so it's per-device and resets on sign-out/cookie clear. Both locales' message files are TypeScript-typed (`messages/en/*.json` is source-of-truth shape; TH must match) so `tsc` fails if a key is missing in either locale, preventing silent drift. The locale switcher is a two-state toggle in the top nav, visible only after sign-in; pre-auth pages (`app/(auth)/...`) are English-only since the switcher lives in `AppShell`.

The reversible-later upgrade paths if needs grow: cookie → URL prefix (when caching or deep-link sharing matters), cookie → `users.locale` DB column (when cross-device sync matters), chrome-only → full translation (when the team adds Thai canonical terms to `CONTEXT.md`). None of the v1 choices lock those out.
