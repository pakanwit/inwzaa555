# Plan: i18n — English + Thai (chrome-only)

**Date:** 2026-05-27 (target ship)
**Related ADRs:** 0008 (i18n is chrome-only across EN and TH — this plan implements the ADR)
**Context terms:** none translated; chrome-only scope. The domain glossary in `CONTEXT.md` is unchanged.

## Goal

Add Thai as a selectable UI language alongside English. After sign-in, members can flip between EN and TH via a toggle in the top nav; the choice persists per-device via cookie. All UI chrome (nav, buttons, form labels, empty states, page titles, loading/empty/not-found copy) renders in the chosen locale. Domain terms, errors, currency, and dates are untouched in v1 — see ADR 0008 for the rationale of each narrowing.

Out of scope (deferred):
- Translating form validation errors and server-action error strings (still English).
- Locale-aware currency and date formatting (`formatBaht` and `toLocaleDateString()` unchanged).
- Translating Supabase auth emails, `app/auth/callback/route.ts` redirect messages, or anything outside the Next.js render tree.
- A `users.locale` DB column / cross-device sync.
- URL-prefixed locale routing (`/th/...`).
- Pre-auth language switching — sign-in / sign-up pages are English-only.

## Decisions locked

| Concern | Decision |
|---|---|
| Library | `next-intl` (App Router + Server Components support, fits Next 16 + `proxy.ts` shape) |
| Routing | Cookie-only (`NEXT_LOCALE`). No `app/[locale]/` refactor. |
| Default | `en` when no cookie set. No Accept-Language sniff. |
| Persistence | Cookie only. No DB column. |
| Domain terms | English in both locales (Pot, Slip, Receipt, Fronted Expense, Fair Share, Net Position, Member, Admin, Contribution, Reimbursed). |
| Errors | English in both locales (Zod messages + server-action `{ ok: false, error }` strings). |
| Formatting | `formatBaht` unchanged; dates remain `toLocaleDateString()` (browser locale). |
| Messages shape | `messages/{en,th}/{common,nav,dashboard,expenses,contributions,members,auth}.json`. Namespace per feature. |
| Parity discipline | TypeScript `Messages` augmentation; `tsc` fails if any TH key is missing relative to EN. |
| Switcher UI | Two-state toggle in `AppShell` top nav, after "Signed in: name". Y2K bevel buttons. |
| Switcher mechanism | Server Action `setLocale(locale)` writes `NEXT_LOCALE` cookie + `revalidatePath('/', 'layout')`. No `proxy.ts` change. |
| Pre-auth pages | English-only. Switcher lives in `AppShell`, not `(auth)`. |

## Implementation steps

### 1. Install and base config

- `npm i next-intl`.
- Create `i18n/request.ts` exporting `getRequestConfig` that reads the `NEXT_LOCALE` cookie (falls back to `'en'`) and loads the namespace files for that locale.
- Create `next-intl.d.ts` augmenting `IntlMessages` with the inferred shape of `messages/en/*.json` so `useTranslations('expenses').(...)` is type-checked end-to-end.
- Add the `next-intl` plugin to `next.config.mjs` (`createNextIntlPlugin('./i18n/request.ts')`).
- **Do not** add the legacy `i18n` config key to `next.config.mjs` — it's Pages-Router-only and a no-op in App Router.

### 2. Provider wiring in the root layout

- Wrap `app/layout.tsx`'s children in `<NextIntlClientProvider messages={...} locale={...}>`. Use `getLocale()` + `getMessages()` from `next-intl/server`.
- Critical: this is **`app/layout.tsx`** (the root), **not** `app/(app)/layout.tsx`. The provider must wrap auth pages too so any client component anywhere in the tree can call `useTranslations`. Pre-auth pages still render in EN (per ADR), but the provider still wraps them so no `useTranslations` call throws.

### 3. Messages files — EN first

Create `messages/en/{common,nav,dashboard,expenses,contributions,members,auth}.json`. EN is the source-of-truth shape — extract every visible English string from these locations into the appropriate namespace file:

- `messages/en/common.json` — shared buttons, generic labels: `save`, `cancel`, `delete`, `edit`, `restore`, `back`, `confirm`, `loading`, `empty`, `viewSlip`, `viewReceipt`.
- `messages/en/nav.json` — nav labels in `components/app-shell.tsx`: `home`, `expenses`, `pot`, `members`, `signedInAs`, `signOut`, and `localeToggle.en` / `localeToggle.th` for the switcher.
- `messages/en/dashboard.json` — `app/(app)/page.tsx` stat labels (`potTotal`, `potRemaining`, `spentSoFar`, `fairSharePerPerson`), section headings, "who paid" labels — but `Pot`, `Fair Share` remain literal English (per ADR they aren't translated; the *key name* describes role, not the *value*).
- `messages/en/expenses.json` — list page, new-expense form, expense-detail page. Form field labels (`description`, `amountThb`, `category`, `date`, `paidBy`), `CATEGORY_OPTIONS` labels (`food`, `transport`, `lodging`, `activity`, `other`), empty states.
- `messages/en/contributions.json` — list page, new-contribution form, contribution-row labels.
- `messages/en/members.json` — list, role badges, action buttons, dialog copy.
- `messages/en/auth.json` — sign-in / sign-up / callback strings. Even though pre-auth is EN-only in v1, keys still belong here so the namespace exists for future TH coverage.

### 4. Messages files — TH parity

Create `messages/th/*.json` mirroring the EN structure exactly (same keys, Thai values). `tsc` enforces this once the type augmentation is in place. Notes for translators (informal — `pakanwit@mohara.co` is doing the translations):

- **Domain terms stay English even inside Thai sentences.** "เพิ่ม Expense ใหม่" not "เพิ่มรายการใช้จ่ายใหม่". "ดูสลิป" is OK (สลิป is a Thai loanword), but `Slip` as a noun in headings stays English: "Slip ของ Contribution นี้". This is intentional; ADR 0008 governs.
- Currency values: don't try to format inside translations — `formatBaht` does it. Translations get the formatted string interpolated: `"จ่ายไป {amount} แล้ว"`.
- Dates: same — translations interpolate the already-formatted date.

### 5. String replacement in components

For each file below, replace the literal string with `t('namespace.key')` (server components: `import { getTranslations } from 'next-intl/server'`; client components: `import { useTranslations } from 'next-intl'`):

- `components/app-shell.tsx` — NAV array labels, "Signed in: X" (becomes `t('nav.signedInAs', { name: currentUser.displayName })`), the Sign-out button text. Add the locale toggle component (next step).
- `app/(app)/page.tsx` — every JSX string in the dashboard. Stat tiles, section headings, list empty states.
- `app/(app)/expenses/page.tsx`, `app/(app)/expenses/expenses-client.tsx`, `app/(app)/expenses/new/page.tsx`, `app/(app)/expenses/new/new-expense-form.tsx`, `app/(app)/expenses/[id]/expense-detail.tsx`.
- `app/(app)/contributions/page.tsx`, `app/(app)/contributions/new/page.tsx`, `app/(app)/contributions/new/new-contribution-form.tsx`.
- `app/(app)/members/page.tsx`, `app/(app)/members/members-client.tsx`.
- `components/features/expense-row.tsx`, `components/features/contribution-row.tsx`.
- `app/not-found.tsx`, `app/(app)/loading.tsx`.
- `lib/expense-form.ts` — `CATEGORY_OPTIONS` labels become keys, not literals: `[{ value: 'food', labelKey: 'expenses.category.food' }, ...]`. The form component looks up `t(opt.labelKey)`. (Alternative: keep labels in JSON only and drop them from the schema file. Either is fine.)

Strings that **stay literal** (per ADR):
- All `.message` strings in `lib/expense-form.ts` and other Zod schemas.
- All `{ ok: false, error: '...' }` strings in `lib/actions/*.ts`.
- `formatBaht` output.
- Date rendering via `toLocaleDateString()`.
- `app/(auth)/...` page chrome (sign-in / sign-up forms) — left in EN, no `useTranslations` calls added.

### 6. Locale switcher

- New component `components/locale-toggle.tsx` (client component). Two bevel buttons; the active one looks pressed (`bevel-in` per y2k conventions). On click, calls the server action `setLocale('en' | 'th')`.
- New server action `lib/actions/locale.ts`:
  ```ts
  'use server'
  import { cookies } from 'next/headers'
  import { revalidatePath } from 'next/cache'

  export async function setLocale(locale: 'en' | 'th') {
    if (locale !== 'en' && locale !== 'th') return
    ;(await cookies()).set('NEXT_LOCALE', locale, {
      path: '/',
      maxAge: 60 * 60 * 24 * 365,
      sameSite: 'lax',
    })
    revalidatePath('/', 'layout')
  }
  ```
- Mount `<LocaleToggle current={locale} />` in `AppShell` between the nav links and "Signed in: name". Compute `locale` in `app/(app)/layout.tsx` via `getLocale()` from `next-intl/server` and pass into `AppShell`.

### 7. Test wiring

- Create `vitest.setup.ts` addition (or a `test-utils/render.tsx` helper) that wraps RTL's `render` with `<NextIntlClientProvider locale="en" messages={enMessages}>...`. Tests assert against the EN strings since that's the source-of-truth set.
- `components/y2k/*.test.tsx` — no change needed; those components don't call `useTranslations`.
- Feature component tests (`components/features/*`, if/when added) — use the wrapper.

### 8. Verification

- `npm run test` — green.
- `npx tsc --noEmit` — green, including the messages type augmentation (delete a key from `messages/th/expenses.json` to confirm it fails, then restore).
- `npm run build` — green.
- Manual smoke:
  - Fresh browser (no cookie): sign in → top nav shows `[EN] TH`, page renders English.
  - Click `TH` → page reloads in Thai, toggle now shows `EN [TH]`, cookie `NEXT_LOCALE=th` set.
  - Refresh: still Thai. Open a new tab to the app root: still Thai (cookie persists).
  - Trigger a form validation error (submit an empty expense form) → the error message renders in English even though the page chrome is Thai (expected per ADR).
  - Visit `/login` while signed out → page renders English regardless of any prior cookie (no switcher, no `useTranslations` calls in `(auth)` pages).
  - Sign out → cookie persists (cookie isn't cleared on sign-out; locale is per-device).
